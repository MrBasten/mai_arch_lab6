# Событийная архитектура

## Вариант

Система бронирования отелей. В работе используются три основные сущности: пользователь, отель и бронирование.

## Команды и события

Команда просит систему выполнить действие. Событие показывает, что действие уже произошло.

| Команда | Событие | Кто публикует |
|---|---|---|
| `RegisterUser` | `UserRegistered` | сервис пользователей |
| `CreateHotel` | `HotelCreated` | сервис отелей |
| `CreateBooking` | `BookingCreated` | сервис бронирований |
| `CancelBooking` | `BookingCancelled` | сервис бронирований |

## RabbitMQ

Для передачи событий выбран RabbitMQ. В этой задаче нужно доставлять факты об изменениях между частями системы, поэтому хватает `topic exchange` и устойчивой очереди.

Используются:

- `exchange` `booking.events`;
- очередь `booking.read_model`;
- ключи маршрутизации `user.registered`, `hotel.created`, `booking.created`, `booking.cancelled`.

## Отправители событий

События публикуют части системы, которые меняют данные:

- сервис пользователей публикует `UserRegistered`;
- сервис отелей публикует `HotelCreated`;
- сервис бронирований публикует `BookingCreated` и `BookingCancelled`.

В проекте отправка показана в [`src/app/producer.py`](./src/app/producer.py): файл отправляет один подготовленный набор событий в RabbitMQ.

## Получатели событий

Одни и те же события могут читать разные части системы:

- уведомления;
- аналитика;
- аудит;
- построение представления для чтения.

В работе реализовано построение представления для чтения: [`src/app/consumer.py`](./src/app/consumer.py) читает очередь и обновляет `data/read_model.json`.

## Формат события

Все события отправляются в JSON:

```json
{
  "event_id": "uuid",
  "event_type": "BookingCreated",
  "event_version": 1,
  "occurred_at": "2026-05-18T08:00:00+00:00",
  "producer": "booking-api",
  "trace_id": "uuid",
  "payload": {}
}
```

Поле `event_id` нужно, чтобы одно и то же сообщение не применялось повторно.

## Доставка

Гарантия доставки: `at-least-once`.

Для этого используются:

- устойчивый `exchange`;
- устойчивая очередь;
- сохранение сообщений на диск RabbitMQ;
- ручное подтверждение через `ack`;
- проверка `event_id` перед изменением представления для чтения.

Если обработчик упадет до отправки `ack`, RabbitMQ может отдать сообщение еще раз. Поэтому перед применением события проверяется, был ли такой `event_id` уже сохранен.

## CQRS

Сторона записи принимает команды и публикует события. Сторона чтения получает эти события и собирает отдельное представление для быстрых запросов.

В этой работе представление для чтения хранится в файле:

```text
data/read_model.json
```

Там лежат пользователи, отели, бронирования и список уведомлений.

## Создание бронирования

Сценарий включает:

1. клиент отправляет команду `CreateBooking`;
2. сервис бронирований проверяет пользователя, отель и даты;
3. бронь сохраняется на стороне записи;
4. публикуется событие `BookingCreated`;
5. обработчик чтения добавляет бронь в `data/read_model.json`;
6. сервис уведомлений отправляет подтверждение пользователю.

## Отмена бронирования

Сценарий включает:

1. клиент отправляет команду `CancelBooking`;
2. сервис бронирований меняет статус брони на `cancelled`;
3. публикуется событие `BookingCancelled`;
4. обработчик чтения обновляет статус в `data/read_model.json`;
5. сервис уведомлений отправляет сообщение об отмене.
